    The beginnings of a technical article(How to refactor code from the perspective of a beginner)

    Every programmer knows how to code. From someone in the middle of bootcamp, to the lowest jr developer, to the highest sr dev on the team. Good code is what fuels the industry. But how does one write good code? Often times the code a beginner writes is functional, but not production level. In this article i will attempt to explain how to refactor your code from the perspective of someone who is nearly a complete beginner.

    Functional code is king

    Before even beginning to speak about refactoring, we need to get one thing straight. Functional code is king. If there is a specific way that you KNOW how to solve a problem brought up in your project, then use that solution first. When you are taking part of a bootcamp, you are typically on a tight schedule. Sometimes you just need to get the logic or design working within a time crunch. In this case, it is better to have working code than no code. An instructor can guide you into how to refactor your code into a better, more succinct solution. As a student in a bootcamp especially, you will be on tight deadline to get a presentable project. Having a working project when it is time to present is much more important. After your personal demo day, thats when the refactoring can begin.

    Good, Better, Best

    Most people when refactoring or looking at code in general, use the Good, Better, and Best paradigm. This basically means there are three levels of solutions to your code. Good, which means the solution you made works, but is not typically dry code. I give a direct example of a good solution to a problem below. Better, which is a solution that solves the same problem, but with less WET code. The answer is more succinct and runs better during its operation. Then Best, this is full blown production level code. As a bootcamper you dont need to worry about finding the best solution, you may not even understand why a Best solution works. When you are first learning its just fine to take a good solution, and make it better.

    Begin Small

    When beginning to refactor, you need to think small. often times there is obvious WET code within your project that you already know how to fix, just ran out of time before the deadline hit. Starting with something that you know how to fix is the best possible starting point you can have. I will give a simple example from my own code that was an easy fix, i just couldnt refactor it before the deadline. Using good, better, and best practices as a framework, this refactor is probably in the better catagory. While it still isnt perfect, it is better than the original code i produced.
